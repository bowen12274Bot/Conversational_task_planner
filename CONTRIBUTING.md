# Contributing Guide

## 目的

本文件說明此專案的基本協作方式，讓成員在撰寫文件、調整系統設計與進行功能開發時，能維持一致的工作流程。

## 協作原則

- 以 MVP 範圍為優先，避免功能擴張
- 先更新文件，再進行較大幅度的設計或實作調整
- 若變更會影響專案方向、模組邊界或技術架構，應同步更新 `docs` 內相關文件
- 優先保留簡單、可 demo、可維護的方案

## 文件維護

目前文件分工如下：

- `docs/project_overview.md`
  - 專案主文件，包含背景、目標、MVP 範圍與整體流程
- `docs/module_description.md`
  - 系統模組說明
- `docs/system_design.md`
  - 技術架構與技術選型
- `docs/README.md`
  - 文件索引

若修改內容涉及以下範圍，請同步更新對應文件：

- 專案定位或範圍變更：更新 `docs/project_overview.md`
- 模組內容調整：更新 `docs/module_description.md`
- 技術選型或架構調整：更新 `docs/system_design.md`

## 分支使用

建議使用簡單的分支策略，避免多人協作時直接在主分支上累積未整理的修改。

### 分支原則

- `main`
  - 保持可閱讀、可整理、可 demo 的狀態
- 個人工作分支
  - 每位成員在自己的分支上進行文件或功能修改
- 合併前確認
  - 確認內容與目前 MVP 範圍一致，再合併回 `main`

### 分支命名建議

可依修改類型命名：

- `docs/project-overview-update`
- `docs/system-design-update`
- `feature/task-panel`
- `feature/chat-flow`
- `fix/api-format`

若是個人開發習慣，也可加上姓名縮寫或帳號：

- `ab/docs-project-overview`
- `bc/feature-task-panel`

### 使用建議

- 一個分支盡量只處理一類主題
- 文件修改與功能修改若高度相關，可以放在同一分支
- 若是大幅度改動，建議先開新分支，不要直接修改 `main`
- 合併前先確認是否與其他人的改動衝突

## 建議工作流程

1. 先確認此次修改是否屬於 MVP 範圍
2. 建立對應的工作分支
3. 若會影響規格，先更新文件
4. 再進行實作或內容補充
5. 完成後確認文件與實作內容一致
6. 再合併回主分支

## Commit 建議

提交訊息可盡量簡潔明確，建議描述：

- 修改了什麼
- 影響哪個部分

例如：

- `docs: update project overview`
- `docs: revise system design`
- `feat: add task panel prototype`
- `fix: adjust API response format`

## 變更範圍建議

提交前請盡量確認：

- 不要一次混合太多不相關修改
- 文件改動與功能改動若高度相關，可一起提交
- 若只是整理文件，應避免同時加入大量功能修改

## MVP 優先順序

目前協作時請優先配合以下原則：

- 優先完成能展示核心流程的功能
- 優先支援作業或小專題截止前規劃
- 優先保留對話式更新，而非完整面板編修
- 非必要功能應放入後續擴充，而非直接納入目前範圍

## 問題回報與討論

若遇到以下情況，建議先討論再修改：

- 想變更專案主題或使用情境
- 想新增超出 MVP 的功能
- 想更換主要技術選型
- 想調整資料模型或系統主流程

## 總結

本專案目前仍以文件驅動與 MVP 驗證為主，因此貢獻時應優先維持：

- 文件一致性
- 範圍清晰
- 架構簡潔
- demo 可行性
