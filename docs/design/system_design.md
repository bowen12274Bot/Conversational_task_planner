# 系統設計

更新日期時間：2026-06-16 00:00:00

## 設計定位

本專案目前以 MVP 為目標，因此系統設計的重點不是建立完整可部署的產品架構，而是先完成一個能在本機啟動、可穩定展示核心流程的版本。

本系統的核心價值在於：

- 使用者輸入模糊的作業或小專題目標
- 系統透過 AI 引導提問與任務拆解
- 將規劃結果顯示於互動式面板
- 使用者再透過對話補充需求，由 AI 更新規劃結果

因此，系統設計會以「網站形式的本機 Web 應用」為主，並採用後端為核心、資料庫負責持久化、AI 由後端統一串接的架構。

## 平台形式選擇

### 選擇結果

本專案採用 **網站形式的本機 Web 應用** 作為 MVP 實作方式。

也就是說，系統會以瀏覽器作為操作介面，使用者透過本機開啟 `localhost` 進行操作，而前端、後端與資料庫則在本機環境執行。

### 選擇原因

選擇網站而非原生 App 或桌面 App，主要原因如下：

- 開發成本較低
- 可快速完成對話區與任務面板介面
- 容易整合前端、後端與 AI API
- Demo 時只需在本機啟動系統即可使用
- 未來若需要擴充成正式產品，也較容易延伸

此外，目前專案不考慮部署，也不依賴手機原生功能，例如推播、定位或相機，因此沒有優先製作原生 App 的必要。

## 整體技術架構

本系統採用三層式架構概念：

1. 前端介面層
2. 後端流程控制層
3. 資料持久化層

其中，AI 模組不直接由前端呼叫，而是由後端統一負責串接與管理。

### 架構角色說明

#### 1. 前端介面層

前端負責提供使用者操作介面，包含：

- 目標輸入區
- 對話顯示區
- 任務規劃面板

前端的主要工作是收集使用者輸入、呼叫後端 API、接收規劃結果並更新畫面。

#### 2. 後端流程控制層

後端是本系統的核心，負責：

- 接收前端輸入
- 整理使用者目前狀態與歷史資料
- 控制對話流程
- 呼叫 AI API
- 將 AI 回傳結果轉換為結構化規劃資料
- 回答與既有規劃相關的一般問題
- 寫入資料庫
- 回傳前端顯示

因此，本系統的價值核心不只是資料儲存，而是後端對整個規劃流程的協調與控制。

#### 3. 資料持久化層

資料庫主要負責保存系統中的使用者資訊與規劃資料，使每次互動結果都能被記錄與重複使用。

資料庫在本系統中的角色是持久化，而不是主導整個流程邏輯。

#### 4. AI 服務層

AI 服務層可視為後端中的一個子模組，專門負責：

- 組合 prompt
- 呼叫外部 AI API
- 解析 AI 回應
- 將結果轉成內部可處理的結構

這一層的目的是把 AI 串接細節與一般業務邏輯分開，讓未來更換模型或 API 時，不需要大幅修改整體系統。

## 系統流程設計

系統的基本資料流如下：

1. 使用者在前端輸入作業或小專題需求
2. 前端將輸入送至後端 API
3. 後端讀取資料庫中的既有規劃狀態與相關資訊
4. 後端整理上下文後呼叫 AI API
5. AI 回傳提問或任務拆解結果
6. 後端將結果轉成結構化資料並寫入資料庫
7. 前端取得最新結果後更新對話區與任務面板

若使用者後續再透過對話補充新資訊，系統則重複上述流程，以產生更新後的規劃版本。

## 技術選型

### 前端：Vue 3 + Vite

本專案前端採用 `Vue 3`，並搭配 `Vite` 作為開發工具。

選擇原因如下：

- 開發者已有 Vue 使用經驗，可降低學習成本
- Vue 適合建構互動式介面
- 可有效處理對話區與任務面板這類元件化畫面
- Vite 啟動快速，適合本機開發與 demo

前端主要負責：

- 顯示對話訊息
- 提供輸入欄位
- 呈現任務清單與子任務
- 呼叫後端 API
- 接收並更新畫面資料

### 後端：FastAPI

本專案後端採用 `FastAPI`。

選擇原因如下：

- 適合 API 型系統
- 結構清楚，便於實作流程控制
- 與 Python 生態整合方便
- 串接 AI API 較自然
- 開發與測試效率高

後端主要負責：

- 管理 API 路由
- 處理輸入資料
- 控制 AI 呼叫流程
- 整理任務規劃資料
- 存取資料庫

### 資料庫：SQLite

本專案資料庫採用 `SQLite`。

選擇原因如下：

- 適合本機執行的 MVP
- 不需要額外安裝或管理資料庫服務
- 啟動成本低
- 足以支援目前單機 demo 的需求

雖然 `SQLite` 不適合高併發或正式大規模部署，但對目前專案階段而言是最符合成本效益的選擇。

### ORM：SQLAlchemy

資料庫操作可使用 `SQLAlchemy` 來管理資料模型與查詢邏輯。

選擇原因如下：

- 可讓資料表結構與 Python 程式碼對應
- 降低直接撰寫 SQL 的負擔
- 若未來需要更換資料庫，較容易延伸

## 資料設計方向

雖然目前尚未進入詳細資料表設計，但依照系統需求，資料庫至少應考慮保存以下幾類資料：

- 使用者資料
- 規劃專案或任務資料
- 對話紀錄
- AI 產生的任務規劃結果
- 更新後的規劃版本

可進一步發展成下列資料表方向：

- `users`
- `plans`
- `chat_messages`
- `tasks`
- `plan_versions`

這樣的設計有助於保留規劃歷程，也方便後續支援多次更新與版本追蹤。

## AI 串接設計

本專案在 MVP 階段將採用外部 AI API 進行功能實作，並以免費或低成本 API 為優先考量。

### 設計原則

- 前端不直接呼叫 AI API
- 所有 AI 呼叫統一經由後端處理
- 後端以 service layer 包裝 AI 串接邏輯
- AI 回傳結果需轉為系統可處理的結構化資料

### 後端 AI service layer 可負責的工作

- 產生引導提問
- 產生初步任務規劃
- 根據新資訊更新既有規劃
- 針對既有規劃產生一般問答或執行建議

例如可抽象成下列內部功能：

- `generate_questions(input)`
- `generate_plan(context)`
- `revise_plan(context, feedback)`
- `answer_plan_question(context, question)`

這樣的設計可以避免整個系統與特定 AI 供應商綁死，也方便未來更換 API 或模型。

## MVP 與完整版本的技術邊界

### MVP 階段

MVP 階段的技術目標如下：

- 可在本機成功啟動
- 可完成基本前後端互動
- 可記錄資料至資料庫
- 可串接 AI API
- 可展示對話區與任務規劃面板
- 可透過對話要求 AI 更新規劃

### 暫不優先處理的項目

以下內容暫不納入目前的系統設計重點：

- 雲端部署
- 使用者登入與權限管理
- 多人協作
- 即時同步
- 複雜通知機制
- 手機原生功能整合
- 精準到時段的排程演算法

## 總結

本專案目前最合適的技術架構，可整理為以下結論：

- 平台形式：本機執行的 Web 應用
- 前端技術：`Vue 3 + Vite`
- 後端技術：`FastAPI`
- 資料庫：`SQLite`
- 資料存取方式：`SQLAlchemy`
- AI 串接方式：由後端 service layer 統一呼叫外部 AI API

此架構能在維持開發成本較低的前提下，支援本專案 MVP 所需的核心功能，並保留未來擴充為更完整系統的空間。
