# MVP 簡要計畫

更新日期時間：2026-04-17 00:31:00

## 目標

本階段的目標，是將「使用者輸入目標 -> AI 引導提問 -> AI 產生初步規劃 -> 前端面板顯示結果」這條核心流程串接完成，形成一個可運作、可展示的初步原型。

此階段的重點，在於驗證本專案最核心的互動流程是否能成立，也就是系統是否能透過 AI 協助使用者從模糊目標逐步形成較清楚的初步規劃。

## 功能說明

本階段 MVP 主要提供從需求輸入到初步規劃顯示的核心功能，讓使用者能從模糊目標出發，經過 AI 提問與規劃整理後，取得一份可閱讀的初步任務規劃結果。

本階段包含的主要功能如下：

- 需求輸入與 context 整理
  使用者可輸入作業或小專題需求，系統會先整理目前已知資訊，作為後續提問與規劃處理的基礎。
  對應 `Context Engineering Module`

- 關鍵資訊補足與 AI 引導提問
  當資訊不足時，系統會透過少量問題補足規劃所需資訊，並以自然語言方式引導使用者回答。
  對應 `Questioning Module` 與 `Response Module`

- 任務分析與初步規劃生成
  系統會根據需求與補充資訊，產生包含主任務、子任務、粗略估時與優先順序的初步規劃。
  對應 `Planning Module`

- 結構化規劃輸出與前端顯示
  系統會將規劃結果整理為可顯示的結構化內容，並呈現在前端面板中。
  對應 `Output Structuring Module` 與前端介面層

- 基本資料保存
  系統會保存需求、對話與規劃結果，作為後續規劃延續與功能擴充的基礎。
  對應 `Persistence Module`

## 計畫範圍

本階段 MVP 以完成核心流程串接為優先，範圍控制如下：

- 納入需求輸入、AI 提問、初步規劃、結構化顯示與基本資料保存
- 不追求多輪長對話
- 不追求高度精準的完整排程
- 不提供完整面板編修能力
- 以至少一個可穩定展示的完整情境為目標

## 里程碑階段

### 里程碑 1：外部骨架建立

#### 階段目標

先建立系統最外層的基本骨架，讓前端、後端與 AI 串接的入口先形成可操作的基礎，作為後續流程與模組設計的前提。

#### 工作內容

- 建立前端畫面骨架
  先完成需求輸入區、對話顯示區與規劃面板區的基本版面，確認主要畫面區塊與資料承接位置。

- 定義資料契約
  先整理前後端之間的基本 request / response 格式，作為後續 API 與流程設計的共同基準。

- 建立 API 管理骨架
  先規劃基本 API 路由與後端入口，讓前端請求能透過固定接口送入後端流程。

- 完成 AI 模型 API 串接
  先讓系統具備呼叫外部模型的能力，並建立基本的輸入、回應與錯誤處理方式。

#### 階段產出

- 可操作的前端基本畫面
- 基本 request / response 契約
- 可用的後端 API 入口
- 可成功呼叫的 AI 模型串接能力

### 里程碑 2：流程骨架建立

#### 階段目標

在既有外部骨架之上，建立系統的基本處理流程，讓需求輸入後能經過初步整理、流程控制與 AI 回覆，形成一版可運作的純對話流程。

#### 工作內容

- 建立 Context Engineering 架構
  先整理需求輸入、已知資訊與必要上下文，作為後續提問與規劃的基礎。

- 建立控制層邏輯分配
  讓後端能依照既定流程安排需求處理順序，決定何時進行 context 整理、何時呼叫 AI，以及何時回傳結果。

- 建立 Response Module 的最小版本
  先建立一層受控的回覆整理能力，使系統不直接將模型原始輸出回傳前端，而能以較穩定的回覆格式呈現需求確認與追問內容。

- 設計最小資料庫結構
  先建立支援 MVP 所需的最小資料保存能力，使需求、對話與後續結果能被記錄。

- 串接 AI 純對話流程
  讓系統先能完成一版以對話為主的基本互動流程，確認前端、後端、AI、回覆整理與資料保存之間的關係可正常運作。

#### 階段產出

- 可運作的 context 整理流程
- 可控制主要流程的後端骨架
- 具基本回覆整理能力的 Response Module
- 可保存基本需求與對話資料的資料庫結構
- 一版可執行的 AI 純對話流程

### 里程碑 3：規劃結構建立

#### 階段目標

將純對話流程進一步發展為具備結構化規劃能力的系統，使 AI 回應不只停留在文字互動，而能形成可顯示於面板的規劃結果。

#### 工作內容

- 建立 Planning 資料形狀草稿
- 建立 Output 資料形狀草稿
- 收斂 Planning 與 Output 之間的資料介面
- 細化 Questioning / Planning / Output 相關模組
- 細化 Response Module 對規劃結果的回覆能力
- 建立結構化規劃結果的保存方式
- 完成結構化資料的前後端串接
- 完成規劃面板的結構化顯示

#### 階段產出

- 初步可用的規劃資料結構
- 可串接的 Planning / Output 資料介面
- 可顯示的結構化規劃結果
- 可配合規劃結果回覆的 Response Module
- 可保存的結構化規劃資料
- 從對話流程延伸出的規劃面板能力

## 驗證標準

本階段 MVP 是否成立，主要可依據以下標準進行驗證：

### 1. 核心流程可完整串接

系統應能完成「使用者輸入目標 -> AI 引導提問 -> AI 產生初步規劃 -> 前端面板顯示結果」這條主流程。

### 2. AI 能提出少量且具規劃價值的補充問題

當使用者輸入資訊不足時，系統應能提出少量關鍵問題，以補足規劃所需資訊。

### 3. AI 能產生具基本可讀性的初步規劃

系統產生的結果應至少包含主任務、子任務、粗略估時與優先順序，且內容需具備基本可讀性，足以讓使用者理解初步規劃方向。

### 4. 前端能清楚呈現結構化規劃結果

系統產生的規劃內容應能在前端穩定顯示，使使用者能辨識任務結構、工作層次與主要安排。

### 5. 系統可穩定展示作業或小專題的情境

本階段應至少完成一個從輸入、提問、規劃到顯示結果的完整 demo 情境，作為 MVP 成立的直接證明。

### 6. 基本資料保存能力可運作

系統應能保存需求、對話與規劃結果，作為後續規劃延續、結果查詢與功能擴充的基礎。
