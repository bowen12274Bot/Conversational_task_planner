# 系統模組說明

更新日期時間：2026-06-16 00:00:00

## 核心功能模組

本系統可細分為七個核心處理模組與一個資料持久化層：

1. Context Engineering Module
2. Response Module
3. Questioning Module
4. Chat Module
5. Planning Module
6. Output Structuring Module
7. Persistence Module

---

## 模組之間的整體關係

本系統的模組運作流程，主要可整理為以下順序：

1. 使用者輸入需求
2. Context Engineering Module 整理需求與上下文
3. Questioning Module 判斷是否需補足關鍵資訊，或是否可進入下一條路線
4. 若本輪是一般規劃相關提問，Chat Module 產生回答內容
5. 若本輪需要建立或修改排程，Planning Module 根據上下文進行初始規劃或修改規劃分析
6. Output Structuring Module 將規劃分析結果整理為結構化排程資料
7. Response Module 產生或包裝對使用者的自然語言回覆
8. Persistence Module 保存需求、對話與規劃結果，供後續流程重複使用

其中，Persistence Module 並非只在流程最後才參與，而是作為整個系統的持久化支援層，持續提供歷史紀錄與既有規劃資料，讓 Context Engineering Module、Chat Module 與 Planning Module 能在後續互動中取得必要脈絡。

---

## 模組一：Context Engineering Module

### 模組目的

此模組的主要功能，是將使用者輸入的原始需求整理成系統可處理的上下文資料，作為後續 AI 回覆、追問、任務拆解與規劃生成的基礎。

由於本系統支援從模糊目標開始規劃，因此使用者一開始提供的資訊往往是零散且不完整的。此模組的重點，不是要求輸入完全結構化，而是先將原始內容轉換為較清楚的背景脈絡。

### 功能說明

當使用者輸入作業或小專題需求後，此模組會先整理目前已知資訊，例如：

- 任務或目標內容
- 截止時間
- 目前進度
- 使用者能力範圍或已知困難點
- 已知限制條件

若系統中已存在相關歷史紀錄，此模組也可透過 Persistence Module 取得既有內容，一併納入上下文整理，使後續模組不只依賴當前單次輸入，而能參考較完整的脈絡。現階段此歷史內容可由 `conversation_history_text` 直接提供。

此模組的設計重點，在於將原始輸入轉為後續流程可重複使用的基礎資料，而不是直接進行任務拆解或對使用者提問。

### 可處理的資訊

- 使用者原始需求文字
- 截止日期或剩餘時間
- 目前進度
- 使用者能力範圍與困難點
- 已知限制與補充條件
- 歷史對話內容與既有規劃摘要

### 模組輸出

- 一份整理後的任務上下文資料，提供 Questioning Module、Chat Module、Planning Module 與 Response Module 使用

---

## 模組二：Response Module

### 模組目的

此模組的主要功能，是負責產生系統面向使用者的自然語言回覆，使整個互動流程不只是資料處理，而能以較自然、清楚的方式與使用者溝通。

### 功能說明

本系統中的 AI 回覆，不只存在於提問階段，也可能出現在需求接收、規劃完成、修改完成或狀態說明等不同情境。因此，回覆行為不應完全依附於追問流程，而應由獨立模組處理。

此模組可負責產生下列類型的回覆：

- 對使用者需求的初步理解與確認
- 追問前的引導文字
- 初步規劃完成後的摘要說明
- 修改完成後的調整說明
- Chat Module 回答結果的對話區文字包裝
- 對目前規劃狀態的簡短回應

此模組可參考 Context Engineering Module 整理出的上下文、Questioning Module 的提問結果、Chat Module 的回答結果，以及 Output Structuring Module 所整理出的最終規劃內容，產生適合顯示於對話區的說明文字。

Response Module 不負責產生規劃相關問題的實質建議內容。若本輪是 `chat` 情境，實質回答應由 Chat Module 產生，Response Module 僅負責整理為前端可呈現的文字。

### 回覆類型

- 需求確認回覆
- 引導說明回覆
- 規劃摘要回覆
- 修改結果回覆
- Chat 回答包裝回覆
- 狀態說明回覆

### 模組輸出

- 一段適合顯示於對話區的自然語言回覆內容

---

## 模組三：Questioning Module

### 模組目的

此模組的主要功能，是在系統判斷目前資訊不足以支撐有效規劃時，由 AI 主動提出少量高價值問題，以補足後續任務拆解所需的關鍵資訊。

### 功能說明

此模組的重點，不是建立一份「必要欄位清單」要求使用者逐一回答，而是判斷目前缺少哪些資訊會顯著影響規劃品質，並優先補齊最重要的部分。

換句話說，本模組的本質是猜測使用者可能遺漏了哪些關鍵資訊，而不是單純要求完整填答。

因此，提問的設計應以少量、聚焦、具規劃價值為原則，避免對話過度發散或造成使用者負擔。問題內容可能聚焦於：

- 截止時間是否明確
- 目前已完成到哪裡
- 哪一部分最困難
- 每天大約可投入多少時間
- 是否已有已知步驟或限制

若目前資訊已足夠支撐後續規劃，本模組也可直接給出「可進入規劃」的判斷結果，而不一定必須先追問。

### 提問重點方向

- 哪些資訊缺漏最影響規劃
- 哪些內容需要向使用者確認
- 哪些資訊可由系統暫時合理推定
- 目前是否已足以進入任務拆解

### 模組輸出

- 一組少量且具方向性的追問內容，或資訊已足夠時的「可直接進入規劃」判斷結果

## 模組四：Chat Module

### 模組目的

此模組的主要功能，是在不修改排程資料的前提下，回答使用者針對目前任務、既有規劃、執行方法或學習策略所提出的一般問題。

本系統的使用者不一定熟悉如何規劃任務，因此除了建立與修改排程之外，也需要能透過自然語言詢問「為什麼這樣安排」、「某階段該怎麼做」、「有哪些練習方向」或「是否有可參考資源」。Chat Module 即負責承接這類與規劃相關但不要求更新排程的需求。

### 功能說明

當 Context Engineering Module 判斷本輪輸入屬於 `chat` 意圖，且 Questioning Module 判斷目前資訊足以回答時，控制層可將資料轉交給 Chat Module。

Chat Module 可根據使用者本輪問題、整理後上下文、既有規劃摘要與必要歷史對話，產生一份固定 JSON 格式的回答結果。此回答結果可再交由 Response Module 包裝成對話區文字。

此模組的設計重點，是將「回答規劃相關問題」與「建立或修改排程」分開。只有在使用者明確要求加入、調整、重排或拆細排程時，才應進入 Planning Module 的修改路線。

### 可處理的問題

- 解釋目前規劃的安排理由
- 回答某個階段或子任務的執行方法
- 提供練習方向、學習策略或注意事項
- 提供一般資源方向或準備建議
- 評估目前規劃可能的風險或執行重點

### 模組輸出

- 一份結構化的回答結果，提供 Response Module 包裝為前端對話區文字

---

## 模組五：Planning Module

### 模組目的

此模組是整個系統中的分析核心，主要負責根據使用者需求與既有資訊進行任務分析與拆解，判斷應將任務拆到何種程度，才足以讓使用者開始行動。

### 功能說明

當系統蒐集到足夠資訊後，此模組會根據目標內容、截止壓力、目前進度、困難點與其他上下文，分析整體任務的組成與工作層次。

此模組的重點不在於直接產生最後呈現給使用者的規劃格式，而是在於完成下列工作：

- 分析任務目標與限制條件
- 判斷應優先處理的部分
- 拆解大任務為較小步驟
- 評估拆解粒度是否足夠可執行
- 粗略推估各部分的工作量
- 判斷任務間的先後關係與調整方向

此模組本質上是在推估「目前使用者情境下，任務應拆到多細才有助於開始執行」，因此更偏向分析與拆解，而非直接進行輸出排版。

此外，此模組可依據系統目前所處階段，分為兩種處理模式：

- `initial planning mode`
- `revision planning mode`

`initial planning mode` 用於使用者首次提出需求、系統尚未存在既有規劃時。此模式的任務，是根據目前蒐集到的需求與上下文，建立第一版任務拆解結果。

`revision planning mode` 用於系統已存在既有規劃，且使用者提出補充資訊、條件變更或修改要求時。此模式的任務，是根據既有規劃、歷史紀錄與新的輸入，判斷應如何調整、重拆或局部更新原有規劃。

在 revision planning mode 下，本模組可透過 Persistence Module 取得目前排程結構與關鍵歷史紀錄，作為修改分析的依據。

Planning Module 不處理一般 `chat` 問答。若使用者只是詢問建議、原因、資源或執行方向，且未要求更新排程資料，應由 Chat Module 承接。

### 分析結果可能包含

- 主任務候選項目
- 子任務拆解結果
- 任務間先後關係
- 粗略工作量判斷
- 需優先處理的工作部分
- 修改模式下的保留、調整或重拆建議

### 模組輸出

- 一份經分析與拆解後的中介規劃資料，提供 Output Structuring Module 使用

---

## 模組六：Output Structuring Module

### 模組目的

此模組的主要功能，是將 Planning Module 產生的分析與拆解結果，整理成系統內部固定可用的結構化輸出，供前端顯示、資料保存與後續修改流程使用。

### 功能說明

雖然任務拆解與輸出整理彼此關聯很高，但兩者的責任不同。Planning Module 著重於分析與拆解；本模組則負責將這些結果組裝成一致、清楚、可呈現的結構化格式。

此模組接收的不是原始需求，也不是直接讀取資料庫內容，而是 Planning Module 已完成判斷後的中介規劃資料。也就是說，Planning Module 先決定「任務怎麼拆、順序怎麼排、估時怎麼抓」；本模組再負責將這些結果整理成固定 schema。

此模組可負責整理的內容包括：

- 主任務名稱
- 子任務階層
- 粗略估時
- 優先順序
- 下一步建議
- 必要的補充說明

這樣的設計可使系統輸出更穩定，也有利於前端顯示、資料保存與後續規劃修改。

### 結構化輸出可能包含

- 規劃標題或主目標
- 任務清單與階層
- 每項任務的估時資訊
- 任務優先順序
- 系統建議的起始步驟
- 修改模式下的更新摘要

### 模組輸出

- 一份結構化的任務規劃資料，提供前端介面顯示、Response Module 產生說明文字，以及 Persistence Module 保存

## 模組七：Persistence Module (已升格為Persistence Layer)

### 模組目的

此名稱原本用於描述系統中的資料保存責任；但依目前架構收斂結果，相關責任已不再只適合以單一模組理解，而是已提升為整體系統中的資料持久化層。

### 架構定位

本系統若要支援對話延續、歷史查詢、規劃修改與多輪互動，就不能只依賴前端暫存或單次對話內容。因此，原本以 `Persistence Module` 描述的資料保存責任，現在已進一步收斂為系統中的 `Persistence Layer`。

目前在文件中仍保留 `Persistence Module` 這個標題，主要是為了延續原有模組閱讀脈絡；但在實際架構理解上，它不應再被視為與 Context Engineering、Questioning、Chat、Planning、Output Structuring 或 Response 並列的處理模組，而應明確理解為一個獨立的架構層。

換句話說，後續若從系統分層角度討論其責任，應優先使用 `Persistence Layer` 這個說法；`Persistence Module` 則只保留作為本文件沿用舊命名時的閱讀入口。

### 功能說明

目前此層主要負責下列工作：

- 建立對話識別與對話主體資料
- 保存使用者端與 AI 端的對話紀錄
- 提供歷史對話資料查詢
- 整理並回傳前端可讀的歷史對話資料
- 保存規劃相關資料與後續版本延續所需內容

此外，此層也負責承接控制層轉交的資料保存需求，並在後續互動中提供既有規劃狀態、關鍵歷史紀錄與必要上下文資料，使 Context Engineering Module 與 Planning Module 能取得後續流程所需的脈絡。

### 可保存的資料

- 任務需求紀錄
- 對話訊息紀錄
- 結構化規劃資料
- 規劃版本資料
- 供後續模組直接使用的完整歷史文字內容

### 模組輸出

- 可供控制層與後續流程讀取、查詢與更新的本地持久化資料
